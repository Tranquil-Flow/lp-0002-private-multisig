# LP-0002 Solution: Private M-of-N Multisig

## Repository

Public repository URL: https://github.com/Tranquil-Flow/lp-0002-private-multisig

## Approach

This solution implements a private threshold approval system where shielded
LEZ account holders can approve proposals without revealing their identity.
The architecture is:

1. **Member secrets** are derived from shielded account material and produce
   deterministic but private commitments.

2. **Nullifiers** bind each approval to `(member, multisig, proposal)` while
   preventing double-voting and maintaining cross-proposal unlinkability.

3. **Threshold proofs** attest that M distinct members approved, without
   revealing which ones. The public journal contains only nullifiers, counts,
   and binding hashes.

4. **Verifier gate** checks the proof journal and executes the proposal exactly
   once, preventing replay.

The implementation is split into a Rust workspace with safe-lane SDK crates plus heavy-lane RISC0/LEZ evidence crates:

- `core/`: Cryptographic primitives, threshold relation, nullifiers
- `verifier-program/`: LEZ-shaped execution gate
- `sdk/`: High-level integration SDK with `MultisigSession`
- `consumer-demo/`: Standalone integration demo (5 scenarios)
- `methods/` + `methods/guest/`: RISC0 threshold-proof guest and executable wrapper guest
- `host/`: proof artifact verification, LEZ bridge, SPEL/NSSA payload evidence, localnet submitter
- `lez-program/`: LEZ-shaped replay-protected account-state wrapper

Plus a browser-facing GUI walkthrough with full SHA-256 crypto in JavaScript, a native Qt/QML Logos Basecamp plugin package, a SPEL IDL, and recording-safe heavy-lane evidence scripts.

## Key Design Decisions

1. **Decoupled from LEZ account model**: Members keep shielded accounts private;
   the verifier only consumes proof journals. This avoids the nonce/program_owner
   constraint that makes the public lez-multisig incompatible with shielded accounts.

2. **SHA-256 domain-separated hashing**: All objects use length-prefixed domain
   separation for collision resistance across different object types.

3. **Sorted nullifiers in journal**: Canonical ordering enables deterministic
   verification and prevents enumeration-based linking attacks.

4. **Resumable partial approvals**: The `ApprovalAccumulator` is serializable via
   serde, enabling client restart without losing collected approvals.

## FURPS Self-Assessment

- **Functionality**: Full threshold relation, nullifiers, context binding,
  consumer integration, real RISC0_DEV_MODE=0 proof artifacts, and confirmed
  compact-wrapper LEZ localnet inclusion evidence. The LP-0002 evaluator/public
  testnet target is the lgs/NSSA localnet; see `submission/TESTNET_EVIDENCE.json`.
- **Usability**: Clone-and-run demo, SDK with 5-step workflow, SPEL IDL, browser preview, and native Qt/QML Basecamp plugin package with local build evidence.
- **Reliability**: workspace tests, typed errors, documented failure paths,
  replay-protected verifier state, and confirmed localnet inclusion evidence.
- **Performance**: microsecond-scale safe-lane operations, RISC0 proof artifact
  measurements, compact wrapper payload metrics, and localnet sequencer execution
  timing; payload/account/receipt cost evidence recorded; current LEZ tooling does not expose a stable per-transaction CU counter, and this limitation is documented rather than guessed.
- **Supportability**: clean workspace, reproducible consumer demo, heavy-lane
  evidence demo, comprehensive docs, and strict readiness validator.
